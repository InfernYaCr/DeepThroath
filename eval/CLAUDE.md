# CLAUDE.md — Проект Eval

## Назначение

Универсальный pipeline для оценки качества ответов RAG-систем.
Метрики: **Answer Relevancy** и **Faithfulness** (DeepEval + внешний LLM-судья).
Поддерживает любые модели-судьи через OpenAI, OpenRouter или GigaChat.

## Ключевые файлы

| Файл | Назначение |
|---|---|
| `eval_rag_metrics.py` | Единственный скрипт — весь pipeline |
| `.env` | Конфигурация провайдера и API-ключей (не коммитить) |
| `.env.example` | Шаблон конфигурации |
| `requirements.txt` | Зависимости |
| `docs/architecture.md` | Полная архитектурная документация |
| `results/{timestamp}_{stem}/` | Папка одного прогона |

## Запуск

```bash
# Полный прогон
.venv/bin/python eval_rag_metrics.py <путь_к_файлу.json>

# Перегенерация отчёта без повторного прогона
.venv/bin/python eval_rag_metrics.py --report-only results/<папка_прогона>
```

## Входной JSON

Каждая запись должна содержать:
- `user_query` — вопрос пользователя (обязательно)
- `actual_answer` — ответ модели (обязательно)
- `expected_answer` — эталонный ответ (отображается в отчёте)
- `retrieval_context` — список чанков из KB (нужен для Faithfulness, иначе метрика не считается)

Дополнительные поля `session_id`, `top_k`, `category`, `intent` используются для группировки в отчёте.

## Выходные файлы (в папке прогона)

| Файл | Содержимое |
|---|---|
| `metrics.json` | Детальные результаты по каждой записи |
| `metrics.csv` | То же в табличном виде |
| `report.md` | Читаемый отчёт с группировкой и причинами оценок |
| `checkpoint.json` | Временный файл восстановления (удаляется по завершении) |

## Конфигурация (.env)

```
JUDGE_PROVIDER=openrouter        # openai | openrouter | gigachat
JUDGE_MODEL=openai/gpt-4o-mini   # название модели под выбранный провайдер
OPENROUTER_API_KEY=sk-or-...
OPENAI_API_KEY=sk-...
GIGACHAT_CREDENTIALS=...
```

## Параметры в коде

| Константа | Значение по умолчанию | Описание |
|---|---|---|
| `THRESHOLD` | `0.7` | Порог pass/fail |
| `MAX_WORKERS` | `10` | Параллельных потоков. Снизить до 5 при `429` |

## Важные детали реализации

- Каждый поток создаёт **свои** экземпляры метрик и судьи — DeepEval не thread-safe при sharing
- Чекпоинт пишется после каждой записи — при падении прогон можно продолжить через `--report-only` или повторный запуск той же папки
- Строка `"The score is X because Y"` от DeepEval постобрабатывается в `format_reason()` → `"Оценка X: Y"`
- Судья инструктируется писать `reason` на русском через системный промпт

## Требования

- Python **3.10+** (deepeval использует синтаксис `X | Y` для типов)
- Виртуальное окружение: `.venv/` (Python 3.11 через Homebrew)
