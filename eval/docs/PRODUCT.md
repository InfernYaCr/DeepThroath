# DeepPulse — Product Concept

## Vision

**DeepPulse** превращает разовую проверку качества RAG-системы в непрерывный измеримый мониторинг.

Платформа закрывает разрыв между командами, строящими AI-продукты на базе RAG, и реальными ожиданиями пользователей. Каждое изменение промпта, базы знаний или модели автоматически оценивается по двум ключевым метрикам и порождает читаемый отчёт с конкретными причинами оценок.

---

## Проблема

Команды, разрабатывающие RAG-системы, сталкиваются с несколькими болями:

1. **Нет объективной метрики качества.** "Ответы стали лучше" — не аргумент. Нужны числа.

2. **Ручная проверка не масштабируется.** Оценивать 500 диалогов вручную после каждого обновления базы знаний — нереально.

3. **Непонятно, где именно плохо.** Низкое качество может быть в конкретной категории вопросов или для конкретного интента — ручная проверка не даёт это увидеть.

4. **Нет трассируемости.** После обновления чанков команда не знает: стал RAG лучше или хуже по каждой категории, по каждому top_k.

5. **Нет связи с безопасностью.** Команды оценивают качество и безопасность раздельно, не видя полной картины AI-системы.

---

## Решение

DeepPulse предоставляет:

| Возможность | Ценность |
|-------------|----------|
| Автоматическая оценка через DeepEval | Сотни диалогов за минуты вместо дней ручной работы |
| Две ключевые метрики RAG | Answer Relevancy + Faithfulness — стандарт индустрии |
| Quality Score (0–100) | Один понятный показатель для руководства |
| Группировка по категориям | Видно, где именно проблемы: категория, интент, top_k |
| Trend-анализ по прогонам | Видно, улучшилось ли качество после каждого обновления KB |
| Поддержка нескольких судей | OpenRouter, OpenAI, GigaChat — любая LLM в роли судьи |
| Единый дашборд с DeepThroath | Качество и безопасность AI-системы в одном месте |

## Целевые пользователи

### Основная аудитория

**AI Product Teams** в компаниях, которые:
- Строят чат-боты и ассистентов на базе RAG
- Регулярно обновляют базу знаний
- Работают в регулируемых отраслях: финтех, healthtech, enterprise

**Роли:**
- ML Engineers — интеграция в CI/CD, настройка судей, выбор top_k
- Product Managers — Quality Score как KPI продукта
- QA Engineers — детальный разбор по категориям и интентам
- Data Scientists — сравнение параметров retrieval (top_k, chunk_size)

### Вторичная аудитория

- **Compliance Officers** — документирование качества AI-ответов для аудита
- **AI Consultancies** — оценка качества RAG-систем клиентов
- **Researchers** — бенчмаркинг различных retrieval-стратегий

---

## Ключевые метрики продукта

| Метрика | Описание |
|---------|----------|
| Quality Score | Среднее Answer Relevancy × 100 (0–100) |
| Answer Relevancy | Насколько ответ релевантен вопросу (0.0–1.0) |
| Faithfulness | Подтверждается ли ответ извлечёнными чанками (0.0–1.0) |
| Pass Rate | % ответов выше порога (default: 0.7) |
| Category Breakdown | Детальные метрики по категориям вопросов |
| Cross-run Delta | Изменение Quality Score между прогонами |

## Конкурентная карта

| Продукт | Тип | Ограничения |
|---------|-----|-------------|
| RAGAS | Open-source Python | Нет дашборда, нет клиентских отчётов |
| TruLens | Open-source | Тяжёлая установка, сложная интеграция |
| Arize Phoenix | Platform | Дорого для малых команд |
| LangSmith | SaaS | Привязан к LangChain |
| Ручная оценка | Процесс | Дорого, нерегулярно, не масштабируется |
| **DeepPulse** | **Pipeline** | **Автоматизация + отчётность + unified dashboard** |

## Сценарии использования

### Сценарий 1 — Обновление базы знаний
Команда добавила 500 новых документов в KB. Запускают DeepPulse на тестовом наборе. Quality Score: 78 → 84. Категория "юридические вопросы": Faithfulness упала с 0.91 до 0.73 — документы плохо чанкуются. Конкретная проблема найдена за 10 минут.

### Сценарий 2 — Выбор top_k
Команда сравнивает top_k=5, top_k=10, top_k=20. Три прогона DeepPulse. Тренд-график показывает: top_k=10 оптимален — выше уже нет прироста в Faithfulness, но растёт шум.

### Сценарий 3 — Клиентский аудит
Консультант оценивает RAG-систему клиента. Запускает DeepPulse, скачивает Markdown-отчёт. В отчёте — Quality Score, разбивка по категориям, конкретные примеры плохих ответов с объяснением судьи на русском.

### Сценарий 4 — Единый мониторинг
AI-команда открывает unified_app.py: слева — Security Score из DeepThroath, справа — Quality Score из DeepPulse. Один дашборд отвечает на вопрос: "Наш AI безопасен И отвечает качественно?"

## Roadmap

### v1.0 (текущий)
- Pipeline оценки: Answer Relevancy + Faithfulness
- Поддержка OpenRouter, OpenAI, GigaChat
- YAML-конфигурация судей (`targets.yaml`)
- Checkpoint-восстановление при падении
- Единый дашборд с DeepThroath

### v1.1
- Дополнительные метрики: Contextual Precision, Contextual Recall
- Сравнение нескольких прогонов side-by-side
- Экспорт в PDF (аналог DeepThroath)
- CI/CD GitHub Actions workflow

### v2.0
- Поддержка локальных моделей (Ollama) как судей
- Кастомные метрики через YAML
- Batch-режим: оценка нескольких конфигов за один запуск
- API для интеграции в любой пайплайн

### v3.0
- Continuous monitoring (не только при обновлении KB)
- Автоматические алерты при деградации Quality Score
- Интеграция с Jira / Linear для создания тикетов
- A/B тестирование retrieval-стратегий
