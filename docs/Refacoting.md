План рефакторинга: Переход на FastAPI и подготовка к деплою
Этот документ описывает ключевые архитектурные изменения, необходимые для перевода проекта DeepThroath из стадии скриптов/Streamlit в полноценное, масштабируемое и готовое к деплою веб-приложение.

User Review Required
WARNING

Главное решение, требующее твоего подтверждения: мы удаляем старые Streamlit дашборды (src/dashboard/) и полностью перекладываем визуализацию и работу с ними на новый Next.js фронтенд (web/)? Если Streamlit больше не нужен, его удаление радикально очистит код. [!IMPORTANT] Для длительных LLM-сканирований (Red Teaming) нам понадобится очередь задач или фоновое выполнение. В первой итерации предлагается использовать FastAPI BackgroundTasks + in-memory или файловый сторадж состояний, чтобы не усложнять деплой поднятием Redis (и Celery). Ты согласен на легкий вариант хранения стейтов джобов (например, SQLite)?

Ключевые инсайты: Ты был прав — архитектура сейчас не годится для деплоя (особенно если фронт будет висеть на чем-то вроде Vercel). Самый критичный кусок: web/src/app/api/runner/route.ts и run/route.ts вызывают питонячьи скрипты через child_process.spawn. Это блокирует потоки в Node.js, оставляет мусорные временные файлы (.tmp_api_config...json) и стопроцентно будет завершаться с Gateway Timeout 504 на долгих тестах LLM, так как роуты не могут висеть по 5–10 минут.

Кроме того, есть очевидные дубли (отдельный старый дашборд на Streamlit в src/dashboard, который конкурирует с Next.js) и недостаток строгих контрактов валидации (в Python-бекенде нет Pydantic-схем для REST, а во фронте всё ещё есть куполы any).

Да, именно так! Но тут есть нюанс — всё зависит от того, откуда именно ты их запускаешь.

Давай разберем оба сценария, чтобы было понятно, где кроется боль:

1. Запуск через Web-интерфейс (Next.js) — ❌ Упадёт 100% при деплое
Если ты нажимаешь кнопку на фронтенде и запрос идёт в твой app/api/run/route.ts, то происходит следующее:

Ограничение платформы (Таймаут): Если ты задеплоишь Next.js, например, на Vercel, у их API-маршрутов жёсткий лимит выполнения (10–60 секунд). Модель (особенно если это большая череда RAG/Deepeval проверок) будет думать дольше. Соединение просто разорвется с ошибкой 504 Gateway Timeout. Браузер ничего не получит.
Блокировка потока: В run/route.ts фронтенд тупо "висит" и ждёт pythonProcess.on('close'). Ты не видишь прогресса, нет логов, и любой сбой по сети обрывает всё.
Убийство фонового процесса: В другом твоем файле, runner/route.ts, процесс спавнится и API сразу возвращает status: 200. Локально это сработает. НО на сервере (в Serverless-окружении) как только Next.js отдал HTTP-ответ, он убивает все фоновые процессы. Твои тесты просто внезапно остановятся на 10-м проценте.
2. Запуск локально из терминала (Python CLI) — ✅ Отработает нормально
Если ты зайдёшь в терминал и руками напишешь python scripts/run_redteam.py, то никаких ограничений по времени нет. Python будет крутить оценки deepeval хоть 5 часов, всё запишется штатно (если только сам OpenAI/Anthropic не отвалится по таймауту).

Вывод
Вся твоя архитектурная проблема сводится к тому, чтобы "подружить" долгие скрипты на Python с вебом. Поэтому перенос на FastAPI абсолютно необходим.

Схема будет простой и надёжной:

Next.js отправляет быстрый запрос: "Эй, FastAPI, начни тесты" и мгновенно получает ответ: "Принято, вот твой job_id = 123".
FastAPI спокойно ставит задачу в фон (BackgroundTasks) и начинает крутить Deepeval. Ограничений на время нет.
Next.js раз в 3 секунды спрашивает: "Как дела у job_id 123?" и рисует красивый прогресс-бар пользователю.
