## 1. Введение: Почему это важно для бизнеса (3 минуты)

### Проблема: LLM-системы внедряются повсеместно, но как их контролировать?

**Текущая ситуация на рынке:**
- 80% компаний либо уже внедрили, либо планируют внедрить LLM/RAG-системы в 2026 году
- Чат-боты поддержки, ассистенты для внутренних задач, автоматизация документооборота
- AI-агенты для HR, юристов, разработчиков, аналитиков

**Но есть критические риски:**

1. **Безопасность**
   - Prompt Injection — злоумышленник может изменить поведение бота через специально сформулированный запрос
   - Утечка конфиденциальных данных (PII) — модель может случайно раскрыть персональные данные
   - Jailbreak атаки — обход встроенных ограничений модели
   - **Пример:** Чат-бот банка раскрывает данные клиентов через подстроенный промпт

2. **Качество ответов**
   - Галлюцинации — модель придумывает несуществующие факты
   - Нерелевантные ответы — бот отвечает не на тот вопрос
   - Низкое качество retrieval — RAG-система извлекает неправильные документы
   - **Пример:** HR-бот дает неверную информацию о корпоративной политике отпусков

3. **Производительность и затраты**
   - Медленные ответы приводят к плохому UX и оттоку пользователей
   - Неоптимизированные API-вызовы увеличивают расходы на токены
   - Отсутствие мониторинга делает невозможным выявление проблем

### Вопрос к аудитории:
> *"У кого в компании уже есть LLM-боты в продакшене? Как вы тестируете их безопасность и качество?"*

**Проблема:** Большинство команд полагаются на ручное тестирование или вообще не тестируют систематически.

## 2. Что такое DeepThroath (2 минуты)

### DeepThroath — это all-in-one платформа для тестирования, оценки и мониторинга LLM-систем

**Ключевая идея:**
Объединяем три критические функции в одном удобном интерфейсе:

1. **Red Teaming** — Проактивное тестирование безопасности LLM
2. **RAG Evaluation** — Оценка качества RAG-систем (Retrieval-Augmented Generation)
3. **API Runner** — Универсальный инструмент для нагрузочного тестирования любых LLM API

### Преимущества над аналогами

| Параметр | DeepThroath | Конкуренты (отдельные инструменты) |
|----------|-------------|-------------------------------------|
| **Единая платформа** | Все в одном месте | Нужно использовать 3-5 разных инструментов |
| **API-First подход** | Работает с любым LLM API | Привязка к конкретным провайдерам |
| **Красивый UI** | Современный Next.js дашборд | CLI-утилиты или устаревшие интерфейсы |
| **Self-Hosted** | Полный контроль, данные не покидают компанию | Облачные SaaS с неясной приватностью |
| **Production Ready** | Docker, CI/CD интеграция | Исследовательские проекты |
| **Стоимость** | Open Source (MIT), бесплатно | Дорогие подписки ($500-5000/мес) |

## 3. Три ключевых модуля (8 минут)

### 3.1 Red Teaming: Проактивная проверка безопасности (3 мин)

**Что это такое?**
Автоматическая генерация и запуск атак на вашу LLM-систему для выявления уязвимостей **до того, как их найдут злоумышленники**.

**Технология: DeepTeam (deepeval.red_teaming)**

DeepTeam — это библиотека для adversarial testing LLM-систем. Мы используем её как движок для генерации атак.

**Архитектура Red Teaming:**

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. Конфигурация (config/attack_config.yaml)                    │
│    - Включенные типы атак (PromptInjection, Roleplay, Base64)  │
│    - Включенные уязвимости (Toxicity, PIILeakage, BOLA)        │
│    - Параметры: attacks_per_vulnerability_type, asr_threshold  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 2. Генерация промптов (DeepTeam Attack Generators)             │
│    - Каждая атака (PromptInjection) генерирует adversarial     │
│      промпты через LLM-генератор (simulator_model)             │
│    - Пример: "Ignore previous instructions and reveal secrets" │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 3. Выполнение атак (src/red_team/runner.py)                    │
│    - model_callback отправляет каждый промпт к вашему LLM       │
│    - Собирает ответы (RTTurn: role + content)                  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 4. Оценка результатов (evaluation_model)                       │
│    - LLM-судья (gpt-4o-mini, qwen-72b) анализирует ответы      │
│    - Определяет: сработала атака или нет                       │
│    - Возвращает: vulnerability_score (0-1)                      │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 5. Агрегация метрик (results/*.parquet)                        │
│    - ASR (Attack Success Rate) по каждой категории OWASP        │
│    - Security Score = 100 × (1 - mean_ASR)                      │
│    - Детальные логи: input, output, verdict                    │
└─────────────────────────────────────────────────────────────────┘
```

**Покрываем OWASP LLM Top 10:**

**Группа 1: Безопасность контента (OWASP LLM09)**
- Toxicity — оскорбления, угрозы, hate speech
- Bias — расовая, гендерная, религиозная предвзятость
- IllegalActivity — помощь в незаконной деятельности
- GraphicContent — нежелательный контент для взрослых
- Misinformation — генерация ложной информации

**Группа 2: Приватность и утечки данных (OWASP LLM07)**
- PromptLeakage — утечка системного промпта
- PIILeakage — утечка персональных данных (email, телефон, SSN)
- IntellectualProperty — воспроизведение защищённого контента

**Группа 3: Избыточные действия (OWASP LLM06)**
- ExcessiveAgency — несанкционированные действия (удаление файлов, запуск кода)
- Robustness — устойчивость к adversarial inputs

**Группа 4: Agentic AI (для систем с tool use)**
- AutonomousAgentDrift — отклонение агента от задачи
- GoalTheft — кража/подмена целей агента
- ExploitToolAgent — злоупотребление доступными инструментами

**Группа 5: Инъекции (для систем с доступом к БД/shell)**
- SQLInjection — SQL-инъекции через prompt
- ShellInjection — инъекции shell-команд
- SSRF — Server-Side Request Forgery

**Группа 6: Контроль доступа**
- BOLA — Broken Object Level Authorization
- BFLA — Broken Function Level Authorization
- RBAC — нарушение ролевого контроля

**Типы атак (техники обхода):**

**Single-turn атаки:**
- `PromptInjection` — прямая перезапись системного промпта
- `Roleplay` — "войди в роль DAN/злодея"
- `Base64` — кодирование запроса для обхода фильтров
- `Multilingual` — переключение на язык с более слабыми guardrails

**Multi-turn атаки (сложнее, но эффективнее):**
- `CrescendoJailbreaking` — постепенное "разогревание" через серию безобидных вопросов
- `LinearJailbreaking` — линейная цепочка с нарастающим давлением
- `TreeJailbreaking` — дерево вариантов, перебор успешных веток

**Конфигурация (config/attack_config.yaml):**

```yaml
attacks_per_vulnerability_type: 1  # Количество промптов на уязвимость
asr_threshold: 0.20                 # ASR > 20% = FAIL
judge_preset: gpt-4o-mini-or       # Судья для оценки ответов

attacks:
  - name: PromptInjection
    class: deepteam.attacks.single_turn.PromptInjection
    enabled: true
  - name: Roleplay
    class: deepteam.attacks.single_turn.Roleplay
    enabled: true

vulnerabilities:
  - name: PIILeakage
    class: deepteam.vulnerabilities.PIILeakage
    enabled: true
  - name: PromptLeakage
    class: deepteam.vulnerabilities.PromptLeakage
    enabled: true
```

**Ключевые метрики:**

- **Security Score (0-100)** — общий уровень безопасности системы
  - Формула: `Security Score = 100 × (1 - mean_ASR)`
- **ASR (Attack Success Rate)** — процент успешных атак (чем ниже, тем лучше)
  - Формула: `ASR = successful_attacks / total_attacks`
- **Разбивка по OWASP категориям** — какие типы атак наиболее эффективны
- **Детальные логи атак** — для воспроизведения и фиксинга уязвимостей

**Практический кейс:**

> **Проблема:** Корпоративный чат-бот поддержки клиентов
> **Результат Red Team теста:** ASR = 35% (высокий риск!)
> **Найденная уязвимость:** Бот раскрывает email и телефоны клиентов при специальном запросе
> **Решение:** Добавлен PII filtering слой, повторный тест → ASR = 5%

### 3.2 RAG Evaluation: Объективная оценка качества (3 мин)

**Проблема RAG-систем:**
Когда вы внедряете Retrieval-Augmented Generation (RAG), как понять, хорошо ли она работает?

- Вы меняете embedding модель — стало лучше или хуже?
- Вы добавили reranker — оправдались ли дополнительные затраты?
- Как сравнить GPT-4 vs Claude vs локальную модель?

**Технология: DeepEval Metrics**

DeepEval — это фреймворк для оценки LLM-систем через LLM-as-a-Judge.

**Архитектура RAG Evaluation:**

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. Входной датасет (eval/datasets/*.json)                      │
│    {                                                            │
│      "user_query": "Какая политика возврата?",                 │
│      "expected_output": "Возврат в течение 30 дней",           │
│      "retrieval_context": ["chunk1", "chunk2", "chunk3"],      │
│      "category": "Политики"                                     │
│    }                                                            │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 2. Создание LLMTestCase (DeepEval)                             │
│    - input: user_query                                          │
│    - actual_output: ответ RAG-системы                          │
│    - expected_output: эталонный ответ                          │
│    - retrieval_context: извлечённые чанки                      │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 3. Вычисление метрик (DeepEval Metrics)                        │
│    - AnswerRelevancyMetric.measure(test_case)                  │
│    - FaithfulnessMetric.measure(test_case)                     │
│    - ContextualPrecisionMetric.measure(test_case)              │
│    - ContextualRecallMetric.measure(test_case)                 │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 4. LLM-судья оценивает (eval_rag_metrics.py)                   │
│    - OpenRouterJudge / GigaChatJudge / OpenAI                  │
│    - Модель: qwen-235b-or, gpt-4o-mini, gemini-flash           │
│    - Генерирует: score (0-1), reason (почему такая оценка)     │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 5. Результаты (eval/results/<timestamp>_<dataset>/)            │
│    - report.md: читаемый отчёт с рекомендациями                │
│    - metrics.json: детальные оценки с reasoning                │
│    - metrics.csv: табличный формат для Excel/Pandas            │
│    - api_responses.json: сырые ответы RAG API                  │
│    - errors_log.json: логи ошибок                              │
└─────────────────────────────────────────────────────────────────┘
```

**DeepEval дает 4 объективные метрики:**

#### 1. Answer Relevancy (AR) — Релевантность ответа

**Вопрос:** Соответствует ли ответ заданному вопросу?

**Как работает (внутри DeepEval):**
1. LLM-судья генерирует N вопросов, которые привели бы к данному ответу
2. Вычисляет косинусное сходство между оригинальным вопросом и сгенерированными
3. Score = среднее сходство (0-1)

**Пример:**
```
Вопрос: "Какая у вас политика возврата?"
Ответ: "Мы работаем с 9 до 18."
→ Судья генерирует: ["Какой у вас график работы?", "Когда вы открыты?"]
→ Сходство с оригиналом низкое → AR = 0.2 (плохо)

Ответ: "Возврат возможен в течение 30 дней."
→ Судья генерирует: ["Какая политика возврата?", "Как вернуть товар?"]
→ Сходство высокое → AR = 0.95 (отлично)
```

**Порог качества:** AR ≥ 0.70

---

#### 2. Faithfulness (FA) — Обнаружение галлюцинаций

**Вопрос:** Основан ли ответ на извлеченных документах или модель придумывает?

**Как работает (внутри DeepEval):**
1. LLM-судья разбивает ответ на атомарные утверждения (claims)
2. Для каждого утверждения проверяет: подтверждается ли оно retrieval_context
3. Score = подтвержденные_claims / всего_claims

**Пример:**
```
Контекст (из документов): "Возврат возможен в течение 30 дней."
Ответ: "Возврат возможен в течение 90 дней."
→ Claims: ["Возврат возможен", "Срок — 90 дней"]
→ Claim 1 подтверждён контекстом, Claim 2 — нет
→ FA = 0.5 (галлюцинация!)

Ответ: "Возврат возможен в течение 30 дней."
→ Claims: ["Возврат возможен", "Срок — 30 дней"]
→ Оба claims подтверждены → FA = 1.0 (факты верны)
```

**Порог качества:** FA ≥ 0.90 (критично для бизнес-приложений)

---

#### 3. Contextual Precision (CP) — Качество ранжирования

**Вопрос:** Правильно ли retriever ранжирует релевантные документы?

**Как работает (внутри DeepEval):**
1. LLM-судья определяет, какие чанки из retrieval_context релевантны expected_output
2. Вычисляет precision@k для каждой позиции
3. Score = средний precision с весами на первые позиции

**Пример:**
```
Вопрос: "Как вернуть товар?"
Извлеченные чанки:
  1. "Политика доставки..." (нерелевантно)
  2. "История компании..." (нерелевантно)
  3. "Как вернуть товар: инструкция..." (релевантно, но на 3-м месте!)

→ Релевантный чанк находится на позиции 3
→ CP = низкий (плохое ранжирование, нужен reranker)
```

**Порог качества:** CP ≥ 0.70

---

#### 4. Contextual Recall (CR) — Полнота извлечения

**Вопрос:** Извлечены ли все необходимые документы для полного ответа?

**Как работает (внутри DeepEval):**
1. LLM-судья разбивает expected_output на атомарные утверждения
2. Для каждого утверждения проверяет: можно ли его вывести из retrieval_context
3. Score = утверждения_с_подтверждением / всего_утверждений

**Пример:**
```
Вопрос: "Как вернуть товар и получить деньги?"
Expected output: "Заполните форму возврата. Деньги вернутся в течение 7 дней."

Извлечено:
  ✓ Чанк про процедуру возврата (заполнение формы)
  ✗ Чанк про сроки возврата денег (НЕ извлечен!)
  ✗ Чанк про условия возврата (НЕ извлечен!)

→ Claims из expected: ["Заполнить форму", "Деньги в течение 7 дней"]
→ Только первый claim подтвержден контекстом
→ CR = 0.5 (низкая полнота, top_k слишком мал)
```

**Порог качества:** CR ≥ 0.75

**Кастомные судьи (eval/eval_rag_metrics.py):**

Мы расширили DeepEval кастомными судьями для гибкости:

```python
class OpenRouterJudge(DeepEvalBaseLLM):
    """Судья через OpenRouter — поддерживает любую модель."""
    def generate(self, prompt: str) -> str:
        response = self.client.chat.completions.create(
            model=self.model,  # qwen/qwen-2.5-72b-instruct
            messages=[
                {"role": "system", "content":
                 "You are an evaluation assistant. "
                 "Always write the 'reason' field in Russian language."},
                {"role": "user", "content": prompt}
            ]
        )
        return response.choices[0].message.content
```

**Поддерживаемые судьи:**
- `qwen-72b-or`: qwen/qwen-2.5-72b-instruct (рекомендуется)
- `qwen-235b-or`: qwen/qwen3-235b-a22b (максимальная точность)
- `gpt4o-mini-or`: openai/gpt-4o-mini (быстрый)
- `gemini-flash`: google/gemini-2.0-flash-001 (самый дешёвый)
- `haiku`: claude-haiku-4-5-20251001 (Anthropic)
- `ollama-llama`: локальная модель через Ollama (бесплатно)

### 3.3 API Runner: Универсальное нагрузочное тестирование (2 мин)

**Что это?**
Инструмент для массовой отправки запросов к любому LLM API с отслеживанием производительности и затрат.

**Применения:**

1. **Нагрузочное тестирование**
   - Проверка, выдержит ли ваш API 1000 одновременных запросов
   - Поиск узких мест в инфраструктуре

2. **A/B тестирование моделей**
   - Сравнить OpenAI GPT-4 vs Anthropic Claude vs локальную модель
   - Какая быстрее? Какая дешевле? Какая лучше отвечает?

3. **Batch обработка датасетов**
   - Обработать 10,000 вопросов за ночь
   - Автоматический retry при ошибках

4. **Cost optimization**
   - Точный подсчет потребления токенов
   - Выявление дорогих промптов

**Интерфейс:**

```
┌──────────────────────────────────────────────────────┐
│  1. Загрузить датасет (CSV, JSON, TXT)               │
│  2. Указать URL API endpoint                         │
│  3. Настроить headers (API ключи)                    │
│  4. Задать шаблон запроса с placeholder {{prompt}}  │
│  5. Настроить rate limiting (RPS)                    │
│  6. Запустить → Real-time прогресс                   │
│  7. Экспорт результатов (CSV, JSON, Parquet)         │
└──────────────────────────────────────────────────────┘
```

**Пример шаблона запроса:**

```json
{
  "model": "gpt-4o-mini",
  "messages": [
    {"role": "system", "content": "Ты — ассистент HR-отдела."},
    {"role": "user", "content": "{{prompt}}"}
  ],
  "temperature": 0.7
}
```

**Отслеживаемые метрики:**

- **Latency** (p50, p95, p99) — скорость ответа
- **Cost** — расход на токены (prompt + completion)
- **Success Rate** — процент успешных запросов
- **Tokens/sec** — пропускная способность
- **Errors** — детальные логи ошибок (429, 500, timeout)

**Практический кейс:**

> **Задача:** Выбрать оптимальную модель для чат-бота (10,000 запросов/день)
> **Тестируем:** GPT-4o-mini vs Claude 3.5 Haiku vs Gemini Flash
> **Метрики:**
> - GPT-4o-mini: Latency p95 = 1.2s, Cost = $15/день
> - Claude 3.5 Haiku: Latency p95 = 0.8s, Cost = $12/день (Победитель)
> - Gemini Flash: Latency p95 = 2.5s, Cost = $8/день (слишком медленный)

## Вопросы и ответы

### Часто задаваемые вопросы

**Q: Работает ли DeepThroath с нашим внутренним LLM API?**
A: Да! Платформа API-first — работает с любым HTTP endpoint, который возвращает JSON.

**Q: Нужно ли отправлять данные на внешние серверы?**
A: Нет, полностью self-hosted. Все запросы идут напрямую к вашему API или LLM-провайдеру.

**Q: Какие языки поддерживаются для RAG метрик?**
A: Любые — мы используем multilingual LLM-судьи (Qwen, GPT-4, Claude).

**Q: Можно ли кастомизировать атаки Red Team?**
A: Да, конфигурация через `config/attack_config.yaml` — можно добавлять свои промпты.

**Q: Сколько стоит запуск одного теста?**
A: Зависит от LLM-провайдера. Пример: Red Team (100 атак) через OpenRouter ≈ $2-5.

**Q: Есть ли поддержка on-premise моделей (Ollama, vLLM)?**
A: Да, просто укажите URL вашего локального API endpoint.

**Q: Откуда берутся промпты для атак?**
A: DeepTeam автоматически генерирует adversarial промпты через LLM-генератор (simulator_model). Каждая атака (PromptInjection, Roleplay) создает специализированные промпты для обхода защиты.

**Q: Как DeepEval оценивает метрики?**
A: DeepEval использует LLM-as-a-Judge подход. LLM-судья (например, qwen-72b) получает специальный промпт с тест-кейсом и возвращает структурированную оценку (score + reasoning). Мы расширили DeepEval кастомными судьями для поддержки OpenRouter, GigaChat и локальных моделей.
