# Roadmap: Доведение до Production-стандарта (2026-04-22)

Этот документ описывает план работ по стабилизации, безопасности и масштабированию системы DeepThroath RAG Evaluation.

---

## 🏗️ Блок 1: Инфраструктура и Хранение состояний (Priority: CRITICAL)

### 1. Переход с In-Memory на Redis/PostgreSQL
*   **Что сделать**: Заменить словарь `jobs = {}` в `src/api/main.py` на постоянное хранилище.
*   **Для чего**: Для персистентности данных.
*   **Почему требуется**: Сейчас при любом перезапуске бэкенда (deploy, crash, restart) все данные о текущих и прошлых задачах стираются. Это недопустимо для системы, где задачи могут длиться часами.

### 2. Graceful Shutdown для FastAPI
*   **Что сделать**: Обработать флаги SIGTERM/SIGINT, чтобы сервер не убивал процессы мгновенно.
*   **Для чего**: Чтобы задачи успевали сохранить свой чекпоинт перед выключением.
*   **Почему требуется**: Без этого мы будем терять прогресс вычислений при каждом обновлении кода на сервере.

### 3. Оптимизация Docker (Multi-stage builds)
*   **Что сделать**: Разделить билд-окружение и рантайм в Dockerfile.
*   **Для чего**: Уменьшить размер образа и ускорить деплой.
*   **Почему требуется**: Текущие образы могут весить гигабайты из-за зависимостей (DeepEval, PyTorch и т.д.). Оптимизация сэкономит место и время на CI/CD.

---

## 🛡️ Блок 2: Безопасность и Доступ (Priority: HIGH)

### 4. CORS и Secrets Management
*   **Что сделать**: Настроить белые списки доменов и вынести API ключи в защищенное хранилище (Vault/Secrets).
*   **Для чего**: Предотвратить несанкционированный доступ и утечку ключей.
*   **Почему требуется**: Текущая политика `allow_origins=["*"]` опасна. Утечка ключей OpenAI/DeepSeek напрямую бьет по кошельку.

### 5. API Rate Limiting
*   **Что сделать**: Ограничить количество запросов к API в единицу времени.
*   **Для чего**: Защита от DoS-атак и случайного перерасхода бюджета.
*   **Почему требуется**: Один зацикленный скрипт может потратить весь месячный лимит за 10 минут.

---

## 📈 Блок 3: Качество кода и Мониторинг (Priority: MEDIUM)

### 6. TypeScript Strict Mode & Ruff Fixes
*   **Что сделать**: Включить строгий режим в `tsconfig.json` и исправить все `any` и `undefined` ошибки.
*   **Для чего**: Снижение количества Runtime-ошибок.
*   **Почему требуется**: JS-подобный код во фронтенде ведет к багам, которые сложно ловить на проде.

### 7. GitHub Actions CI/CD
*   **Что сделать**: Автоматизировать запуск `lint.sh` и тестов при каждом Push/PR.
*   **Для чего**: Гарантия того, что в мастер-ветку не попадет сломанный код.
*   **Почему требуется**: Ручной запуск линтера (как мы делали сегодня) — это человеческий фактор, который всегда подводит.

### 8. Мониторинг (Prometheus + Structured Logging)
*   **Что сделать**: Перейти от простых `print()` к JSON-логам с контекстом (Job ID, User ID) и добавить экспорт метрик.
*   **Для чего**: Понимать «здоровье» системы в реальном времени.
*   **Почему требуется**: В логах сейчас каша. Если задача зависнет, мы не поймем почему без глубокого дебага.

---

## 🧪 Блок 4: Тестирование (Priority: ONGOING)

### 9. Pytest & Vitest Integration
*   **Что сделать**: Покрыть основные эндпоинты FastAPI и компоненты UI тестами.
*   **Для чего**: Уверенность при рефакторинге.
*   **Почему требуется**: Система становится сложной (прокси, параллельные потоки). Любое изменение может сломать соседний модуль.

---

## 🦾 Резюме техлида:
Завтрашний фокус — **Блок 1 (Redis + Shutdown)**. Это даст нам фундаментальную стабильность, после которой можно будет спокойно навешивать безопасность и тесты. Без надежного хранения данных всё остальное — просто «украшательство».
